# R12.15 — Pipeline Real de Execução USB (Marlin / Web Serial)

**Status:** ✅ Implementado e testado end-to-end (mock).
**Data:** 2026-05-17
**Sessão:** R12.15 (pivô após R12.14)

---

## H.1 Diagnóstico — O que estava quebrado

Antes desta release, o projeto BIA tinha **dois grupos de código** que NÃO se conectavam:

### Grupo A — Conceitual (R12.14 e anteriores)
- `/dashboard/bioprint/quick-gcode` — gera G-code paramétrico
- `/dashboard/bioprint/gcode/medical` — engine médico de 16 tecidos
- `/dashboard/bioprint/gcode/advanced` — NAATIV3 vector field
- `/dashboard/bioprint/slice` — fatiamento + TissueDesigner (Designer de Tecido Inteligente)

Todos esses geravam G-code válido **mas só ofereciam download** (botão "Baixar .gcode") — o usuário tinha
que abrir Pronterface/OctoPrint para enviar à máquina.

### Grupo B — Execução (existente mas inacessível)
- `/dashboard/bioprint/control` (Etapa 4 do fluxo de 5 etapas)
- `src/components/bioprinting/PrinterConnection.tsx` — **já tinha Web Serial API real** (550 linhas)

**5 problemas que bloqueavam o uso real:**

| # | Problema | Detalhe |
|---|---|---|
| 1 | **Gating de 5 etapas** | `/control` só desbloqueava se `state.slice.status === "ready"`. Forçava Modelo → Biotinta → Slice (com TissueDesigner) → Execução. |
| 2 | **Sem ok-handshake** | `streamGCode` enviava linha + `setTimeout(20ms)` + próxima. **Não esperava `ok` do firmware** — buffer overflow garantido com Marlin real após 4-8 linhas. |
| 3 | **G-codes não chegavam** | `/gcode/medical` e `/gcode/advanced` (R12.14) **não escreviam** em `state.slice.gcode` → impossível enviar pra impressora. |
| 4 | **Sem validador estático** | Nenhuma checagem de G28 inesperado, Z negativo, out-of-volume, temperaturas absurdas, comandos desconhecidos. |
| 5 | **Sem mock simulator** | Impossível testar em sandbox/CI. Web Serial não funciona fora de Chrome/Edge com HTTPS. |

---

## H.2 Plano de fix aplicado

**Plano A + opção (a)**: rota standalone `/execute` sem gating, módulos refatorados em camadas claras, TissueDesigner mantido (mas fora do caminho crítico). Joystick lateral preservado. Preview profissional 3D mantido.

## H.4 Instruções de teste

### Modo MOCK (qualquer browser, qualquer SO)

1. Acesse `/dashboard/bioprint/execute`
2. No header, deixe **Mock** selecionado (default)
3. Painel lateral direito: clique **Iniciar Simulador**
4. Painel esquerdo: clique **Demo** para carregar o Hello Square
5. Clique **Validar G-code** — deve mostrar `✓ Pronto para envio`
6. Verá o **preview 3D** com 2 camadas
7. Clique **Enviar para Bioimpressora**
8. Acompanhe progresso real-time: ~5 segundos
9. Terminal serial vai mostrar tx/rx/ok intercalados

### Modo REAL (Chrome/Edge + bioimpressora física conectada via USB)

1. **Pré-requisitos**:
   - Chrome 89+ ou Edge 89+ (Firefox/Safari NÃO funcionam — Web Serial não suportado)
   - URL com HTTPS ou localhost (contexto seguro)
   - Cabo USB conectando computador → bioimpressora ligada
   - Driver da placa (CH340 / FTDI / CDC) já instalado no sistema

2. **Conectar**:
   - Acesse `/dashboard/bioprint/execute`
   - Header → mude para **Real USB**
   - Painel direito: escolha baud (115200 é padrão Marlin/Ender)
   - Clique **Conectar USB**
   - Diálogo do navegador: selecione a porta serial da impressora
   - Aguarde mensagem `Firmware detectado: marlin · Marlin v2.x.x`

3. **Validar antes de enviar**:
   - Carregue um G-code de teste (recomendado: começar com Demo)
   - Clique **Validar G-code**
   - **NÃO** envie nada com veredito ✗ Bloqueado

4. **Posicionar a impressora**:
   - Use o **joystick lateral** para posicionar cabeçote/bandeja
   - **NÃO existe G28** (home) — bioimpressoras preservam cartucho/bandeja
   - Use **G92 Zero aqui** (botão `⌂` central) para zerar coordenadas no ponto atual

5. **Enviar**:
   - Clique **Enviar para Bioimpressora**
   - Acompanhe linha/camada/% em tempo real no header
   - Use **Pausar** para inspeção (motor + extrusão off na próxima linha)
   - Use **Cancelar** para parar com segurança (footer: M104 S0 + M140 S0 + M141 S0 + M84)
   - Use **EMERGENCY (M112)** APENAS em risco real — trava o firmware e exige restart físico

## H.5 Checklist de validação em bioimpressora real

Use este checklist na PRIMEIRA conexão com uma bioimpressora real:

### Antes de conectar
- [ ] Driver USB da placa instalado (verifique `/dev/ttyUSB*` ou COMx)
- [ ] Bioimpressora ligada e sem erro no display
- [ ] Cartucho carregado (ou removido, se for só teste de movimento)
- [ ] Bandeja livre de obstruções
- [ ] Chrome ou Edge 89+ aberto em HTTPS ou localhost
- [ ] G-code de teste **sem G28** (ou Demo carregado)

### Handshake
- [ ] Web Serial dialog mostra a porta da impressora
- [ ] Conexão sobe (badge verde **ONLINE**)
- [ ] M115 retorna firmware **antes de 5s**
- [ ] Família detectada é `marlin` ou `reprap`
- [ ] Pelo menos algumas Caps são listadas

### Joystick (impressora parada)
- [ ] X+ move o eixo X 1 mm (com step=1)
- [ ] X− volta 1 mm
- [ ] Y+/Y− idem
- [ ] **Z+ sobe** (CUIDADO — confirme direção física)
- [ ] **Z− desce** (CUIDADO COM A BANDEJA)
- [ ] E+ extruda devagar
- [ ] G92 (botão ⌂): coordenadas vão a 0 sem mover

### Stream de teste (Demo Hello Square)
- [ ] Veredito do validador: ✓ Pronto para envio
- [ ] Stream começa, primeira linha = G21
- [ ] Cada linha mostra → no terminal seguido de ←ok
- [ ] Progress bar avança suavemente
- [ ] Pausa → linha atual termina, próxima espera
- [ ] Resume → continua de onde parou
- [ ] Cancel → safety footer roda, motores desligam, hotend → 0

### Emergency stop (TESTAR COM IMPRESSORA SEM CARTUCHO ATIVO)
- [ ] Botão EMERGENCY mostra confirmação
- [ ] M112 enviado, firmware trava
- [ ] Restart físico da placa (ciclo de energia) → impressora volta

### G-code real (após Demo passar)
- [ ] Importe G-code de `/gcode/medical` ou `/gcode/advanced`
- [ ] Verifique bbox cabe no volume da impressora
- [ ] Confirme temperaturas (M104/M140) dentro do limite biológico
- [ ] Inicie stream e observe primeiras camadas

## H.7 Plano wireless (próxima fase)

USB funcionou. Para wireless, três caminhos possíveis (em ordem de simplicidade):

### Opção A — Web Bluetooth (mesma filosofia do Web Serial)
- **Pro**: roda no mesmo browser, sem servidor extra
- **Con**: bioimpressora precisa ter módulo BLE (ESP32 / HM-10 / Marlin com bluetooth_serial)
- **Implementação**: novo módulo `printer-connection-bluetooth.ts` implementando `PrinterTransport`
  → drop-in replacement no Controller (zero código novo no resto da app)
- **Esforço**: ~1 dia

### Opção B — Local Agent + WebSocket (mais robusto, recomendado para produção)
- **Pro**: funciona com qualquer bioimpressora USB existente, sem firmware custom
- **Con**: requer Node.js agent rodando na máquina conectada à impressora
- **Implementação**:
  - Agent: `pkg`-bundled binary 5 MB com `serialport` lib expondo WebSocket `:9999/ws`
  - Browser: novo `PrinterConnectionWS implements PrinterTransport`
  - Auth: token único gerado no setup, copiado para o browser
- **Esforço**: ~3 dias (agent + browser + auth)

### Opção C — Cloud relay (OctoPrint-like)
- **Pro**: acesso remoto de qualquer lugar
- **Con**: requer servidor + endpoint TLS + latência maior
- **Implementação**: agent reportando para Cloudflare Durable Object ou Supabase Realtime
- **Esforço**: ~1 semana

**Recomendação para R13.x**: começar com **Opção A (Bluetooth)** para um experimento rápido, depois **Opção B (Local Agent)** para produção. Opção C é "nice to have" depois.
